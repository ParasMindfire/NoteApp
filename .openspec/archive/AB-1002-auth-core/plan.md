## Prisma Schema Changes

```prisma
model User {
  id            String         @id @default(cuid())
  email         String         @unique @db.VarChar(255)
  passwordHash  String
  createdAt     DateTime       @default(now())
  updatedAt     DateTime       @updatedAt
  refreshTokens RefreshToken[]
}

model RefreshToken {
  id        String    @id @default(cuid())
  userId    String
  token     String    @unique @db.VarChar(64)
  expiresAt DateTime
  revokedAt DateTime?
  createdAt DateTime  @default(now())
  user      User      @relation(fields: [userId], references: [id], onDelete: Cascade)
}
```

- Triggers watcher agents after completion

## Risk Areas

**R1 — Timing side-channel on login (AUTH-LOGIN-S2 vs S3)**
The "unknown email" path skips bcrypt.compare (fast); "wrong password" path
runs it (slow ~100ms). This leaks account existence via response timing even
though the error code and body are identical.
_Mitigation_: when email is not found, call `bcrypt.compare(password, DUMMY_HASH)`
where `DUMMY_HASH` is a pre-computed constant bcrypt hash, then always return 401.

**R2 — Cookie in Supertest (AUTH-REFRESH-S1..S3, AUTH-LOGOUT-S1..S2)**
Supertest does not auto-handle Set-Cookie / Cookie headers between requests.
_Mitigation_: tester agent must extract the `Set-Cookie` value from the login
response and pass it via `.set('Cookie', header)` on refresh/logout requests.

**R3 — Prisma transaction atomicity (AUTH-REFRESH-S1)**
If the revoke and insert are separate Prisma calls, a process crash between
them leaves a revoked token with no replacement.
_Mitigation_: use `prisma.$transaction([update, create])` in the refresh handler.

**R4 — Rate limit state bleeds across tests (AUTH-REGISTER-S4, AUTH-LOGIN-S4)**
`express-rate-limit` default MemoryStore is stateful within a process; tests
making N requests before the rate-limit test will cause premature 429s.
_Mitigation_: the auth router factory accepts an optional `store` parameter;
rate-limit tests inject a fresh `new MemoryStore()` per test.

**R5 — Lockfile must be updated after T3**
The Husky pre-commit hook runs `pnpm install --frozen-lockfile`; committing
`package.json` with `cookie-parser` added but without updating `pnpm-lock.yaml`
will fail the hook.
_Mitigation_: run `pnpm install` (not `--frozen-lockfile`) in T3 to regenerate
the lockfile; commit both files together.
